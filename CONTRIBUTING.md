# 贡献与分支协作规则

本项目是跨平台开发项目，包含 `main`、`mgkccc-win`、`mgkccc-mac`、`mgkccc-linux` 等分支。提交代码前，请先确认当前分支、改动范围，以及改动属于通用内容还是平台专属内容。

## 分支定位

- `main` 是通用代码和通用文档基线，不是相对完整的日常运行目标。
- 日常开发、运行和平台验证应优先在具体平台分支进行，例如 `mgkccc-win`、`mgkccc-mac`、`mgkccc-linux`。
- 平台分支中的通用成果，在测试通过后再同步回 `main`。

## 通用改动与平台专属改动

- 通用改动包括通用 UI、通用状态管理、通用播放/歌词/资料库逻辑、共享类型、通用 IPC 骨架、跨平台接口、通用文档和通用协作规则。
- 平台专属改动包括 Windows/macOS/Linux 各自的系统 API、媒体会话、设备检测、窗口行为、权限、打包脚本、shell 命令和平台后端实现。
- 同步回 `main` 时，只同步通用代码、通用骨架和通用说明；不要把平台专属实现同步回 `main`。

## 需要留意的协作文件

这些根目录文件属于跨分支协作文件：

- `AGENTS.md`
- `bug-list.md`
- `DEVELOPMENT_LOG.md`
- `LICENSE.txt`
- `README.md`
- `CONTRIBUTING.md`

如果更新的是通用规则、通用 BUG、通用待开发功能、通用开发日志、通用许可证说明或通用项目说明，要同步回 `main`。如果只是某个平台专属 BUG、平台专属方向、平台专属日志或平台专属说明，只写入对应平台分支。

## 提交拆分

- 通用代码的 commit 和平台专属代码的 commit 要清晰分开。
- 不要让通用 commit 依赖平台专属 commit。
- 如果在平台分支里先完成了通用改动，应把可复用部分拆成可同步回 `main` 的独立提交。

## 跨平台注意事项

修改通用代码时，不要默认某个系统的 API、路径、窗口行为、音频能力、字体、权限、设备检测、打包方式或 shell 命令在其他系统也可用。通用层应保留平台分支可分别填入实现的接口和骨架。

## push 前检查

完成一次开发后，push 前先总结本次开发内容：

- 如果没有涉及通用改动，只更新平台专属代码和平台专属说明文件。
- 如果涉及通用代码，先把对应说明文档和代码同步回 `main`。
- 同步回 `main` 后，单独测试 `main` 分支能否编译或运行，确认通用骨架不依赖任何平台分支里的专属实现。
- `main` 验证通过后，再把 `main` 同步回平台分支一次。
- 最后再 push。

## AI 助手提醒

很多开发者会通过 AI 助手改代码。AI 助手读到本规则时，必须把分支协作作为高优先级提醒：在用户 push 前，主动提醒用户确认当前分支、改动类型、通用内容是否已同步回 `main`、平台分支是否已重新同步 `main`，以及对应说明文件是否已更新。
