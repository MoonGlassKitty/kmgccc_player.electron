# Linux 分支协作和 PR 清单

这份清单给正在改 Linux 分支的同学看。目标不是限制写法，而是让每次改动更容易被合并、同步和回滚。

## 当前项目大概是什么

mgkccc 是从原 kmgccc 项目转译和继续维护出来的 Electron 版本。现在几个分支会互相同步一部分通用前端改动：

- `main`
- `mgkccc-win`
- `mgkccc-mac`
- `mgkccc-linux`

如果改的是纯 UI、歌词、播放队列、设置页、关于页这类通用逻辑，尽量写得平台无关，后续比较容易同步到其他分支。

如果改的是 Linux 独有能力，比如播放器集成、系统托盘、窗口行为、打包格式、桌面入口，放在 Linux 分支里单独处理即可。

## 开始前

1. 先拉最新远端：

```bash
git fetch origin
git switch mgkccc-linux
git pull --ff-only
```

2. 新建自己的工作分支：

```bash
git switch -c linux/your-change-name
```

分支名不用太正式，能看懂做什么就行。

## 建议优先帮忙看的地方

### Linux 播放源集成

Windows 这边第三方播放源依赖系统媒体控制。Linux 侧可以优先研究：

- MPRIS / DBus 当前播放信息
- 播放/暂停/上一首/下一首控制
- 当前进度和总时长
- 封面 URL 或封面文件读取
- 播放器来源名

如果某个播放器拿不到控制能力，可以让 UI 按钮变灰，不要假装可用。

### Linux 打包

可以检查：

- AppImage 是否能跑
- deb/rpm 是否需要
- 图标是否被正确打进包
- `.desktop` 文件名称、分类、图标是否正确
- 打包后应用名是否是 `mgkccc`

### Linux 窗口行为

可以重点看：

- 无边框窗口拖拽区域是否正确
- 顶部空白拖动窗口时是否会误触最大化
- 全屏歌词和窗口歌词区域能否正常拖拽窗口
- 缩放、最小化、关闭是否符合桌面环境预期

### Linux 性能

可以帮忙观察：

- 第三方源刷新是否导致 1 秒左右顿一下
- 拖动窗口或改变窗口大小时是否卡顿
- 全屏歌词、艺术背景、LED、频谱是否吃性能
- 低配设备是否需要默认降级某些效果

## PR 内容建议

PR 标题尽量一句话说明结果，例如：

- `Fix Linux MPRIS playback state sync`
- `Add Linux AppImage icon packaging`
- `Improve Linux frameless window drag regions`

PR 描述可以按这个模板写：

```md
## 做了什么

- 
- 

## 怎么测试

- [ ] npm run typecheck
- [ ] npm run build
- [ ] Linux 本机启动测试
- [ ] 本地播放测试
- [ ] 第三方播放源测试

## 需要注意

- 
```

不用写得很硬，说明清楚就行。

## 提交前检查

请尽量跑：

```bash
cd windows-player
npm run typecheck
npm run build
```

如果 Linux 分支有自己的打包脚本，也请跑一次对应脚本。

如果某项没法跑，在 PR 里直接写原因，比如：

- 本机没有 Linux 桌面环境。
- 当前依赖缺少某个系统包。
- 第三方播放器无法安装。

## 改代码时的注意点

- 不要提交 `node_modules/`、`out/`、`dist/`。
- 不要把本机临时路径写死进代码。
- 平台判断尽量集中，不要在 UI 各处散落很多 Linux 特判。
- 能复用现有组件就复用，避免同一套设置页或播放器 UI 写两份。
- 涉及播放状态时，注意区分“读取状态”和“控制播放”。
- 外部源拿不到的信息就返回不可用，让 UI 自然变灰。

## 和其他分支同步时

如果改动是通用的，建议在 PR 描述里写一句：

> 这部分应该可以同步到其他分支。

如果改动只适合 Linux，也写清楚：

> 这部分是 Linux/MPRIS/打包专用，不建议同步到其他分支。

这样后面 cherry-pick 或合并时比较省事。

## 可以先做的小任务

- 检查 Linux 打包图标是否不是默认 Electron 图标。
- 检查播放源切到第三方时，不能控制的按钮是否会变灰。
- 检查窗口拖拽区域是否误触最大化。
- 检查播放队列弹窗在 Linux 下能否拖拽排序。
- 检查歌词提前量、字体大小、字重设置是否能保存。
- 检查全屏歌词皮肤切换后颜色是否刷新。

## 遇到不确定的地方

不确定的时候可以先开一个小 PR 或 draft PR，把问题写在 `需要注意` 里。这个项目现在还有不少边改边稳定的地方，先把现象和测试环境写清楚，比一次性猜完整更有用。
