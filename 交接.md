### 实现方式

Windows 主进程优先尝试：

- `@nodert-win11/windows.media.control`
- `@nodert-win11/windows.media`
- `@nodert-win11/windows.storage.streams`

但这些 NodeRT 包是原生模块。macOS 交叉打 Windows 包时无法为 Windows rebuild，所以当前还加了 PowerShell WinRT fallback：

- 使用 `powershell.exe`
- 调 `Windows.Media.Control.GlobalSystemMediaTransportControlsSessionManager`
- 读取 GSMTC snapshot
- 控制 play/pause/next/previous/seek

也就是说，在 Windows 上如果 NodeRT 可用会优先走 NodeRT；如果 NodeRT 因二进制不可用失败，会走 PowerShell fallback。

### 必须在 Windows 上验证

macOS 无法验证 GSMTC。接手后在 Windows 上优先测：

1. 打开 exe。
2. 播放网易云/QQ 音乐/浏览器音频。
3. 侧栏切到 `第三方音乐软件`，看标题、艺人、进度是否显示。
4. 选择 `其他源`，用浏览器播放音频验证。
5. 选择 `自动检测`，验证浏览器和第三方播放器都能被检测。
6. 测底栏：
   - 播放/暂停
   - 上一首/下一首
   - 时间跳转
7. 点本地歌曲，确认能切回本地源，本地 `<audio>` 不会和外部源互相抢状态。

如果外部源不可用，先看主进程控制台/日志是否是：

- NodeRT require 失败后 fallback 没走通。
- PowerShell 语法不兼容 Windows PowerShell 5.1。
- 目标播放器没有注册 GSMTC。

## 打包注意事项

`windows-player/package.json` 里目前：

```json
"package:win": "npm run build && electron-builder --win portable --x64"
```

并且 build 配置里有：

```json
"npmRebuild": false
```

这个不能随便删。原因：

- `@nodert-win11/*` 是原生模块。
- 在 macOS 上打 Windows 包时，electron-builder 会尝试 rebuild。
- node-gyp 不支持从 macOS 交叉编译这些 Windows native 模块。
- 开启 rebuild 会报：`node-gyp does not support cross-compiling native modules from source.`

如果之后在真正 Windows 开发机上打包，可以考虑重新评估：

- 是否允许 `npmRebuild: true`
- 是否能生成真正 Windows x64 的 NodeRT `binding.node`
- 是否保留 PowerShell fallback 作为保险

现在不要删除 fallback。

## 当前已知技术风险

- NodeRT 包现在在 macOS 打包产物里可能包含 mac arm64 的 `binding.node`。运行在 Windows 上会 require 失败，这是预期的，应该 fallback 到 PowerShell。
- 如果 PowerShell fallback 也失败，外部播放源会显示不可用或无数据。
- PowerShell fallback 的性能比常驻 provider 差，当前 renderer 轮询周期是约 700ms，之后可优化为主进程常驻 PowerShell helper 或改用 Windows 原生编译的 NodeRT。
- portable exe 生成阶段可能看起来卡很久，通常是在 7za 压缩 NSIS payload。看 `dist/kmgccc-player-electron-0.1.0-x64.nsis.7z` 是否增长即可判断。
